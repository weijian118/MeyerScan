一、项目背景
本模块是口腔口内扫描设备软件的核心流程控制单元，模块名称为 ScanProcessGenerator。用于根据患者临床修复方案（修复类型、种植场景、扫描杆类型、临时牙状态、备牙需求等参数），自动生成标准化的扫描步骤顺序，控制扫描界面的按钮展示与跳转逻辑，确保临床操作流程规范统一。
开发语言：C++
依赖框架：Qt 5.x
关联模块：扫描界面控件 ScanProcessWidget、系统配置类 ScanConfig
设计原则：上下颌逻辑完全对称、规则可配置、边界条件明确、无隐式逻辑
二、核心功能
动态流程生成：根据输入配置，自动生成上颌、下颌、咬合三个部位的扫描位点顺序，可选生成数据交换步骤
下一位点查询：根据当前选中的扫描位点，查询全流程中的下一个扫描位点（自动跳过数据交换按钮）
规则化控制：所有临床规则通过配置参数控制，无需修改核心逻辑即可适配不同临床场景
三、输入参数定义
3.1 枚举定义
所有枚举均为强语义定义，修复类型支持位或组合使用。
上颌修复类型 EMaxillaType（位或组合）：
Maxilla_None = 0x00，无修复
Maxilla_Repair = 0x01，常规修复
Maxilla_Plant = 0x02，种植修复
Maxilla_DiffRod = 0x04，异性扫描杆
下颌修复类型 EMandibleType（位或组合）：
与上颌完全对称，包含 Mandible_None、Mandible_Repair、Mandible_Plant、Mandible_DiffRod
咬合类型 EOcclusionType：
Occlusion_Natural = 0，自然咬合
Occlusion_Record = 1，咬合记录
注：原「临时牙咬合」已删除，拆分为独立临时牙开关
备牙状态 EBeforePrep：
BeforePrep_No = 0，关闭备牙扫描
BeforePrep_Yes = 1，开启备牙扫描
扫描部位 EScanPartType：
ScanPart_Maxilla，上颌部位
ScanPart_DataExchange，数据交换部位
ScanPart_Mandible，下颌部位
ScanPart_Occlusion，咬合部位
扫描位点 EScanSiteType：
上颌位点：自然牙上颌、备牙上颌、袖口上颌、扫描杆 1 上颌、扫描杆 2 上颌、异性扫描杆 1 上颌、异性扫描杆 2 上颌
下颌位点：与上颌一一对应，包含自然牙下颌、备牙下颌、袖口下颌、扫描杆 1 下颌、扫描杆 2 下颌、异性扫描杆 1 下颌、异性扫描杆 2 下颌
咬合位点：自然牙咬合、咬合记录
交换位点：数据交换按钮
无效位点：ScanSite_None
3.2 配置参数清单（ScanConfig）
所有参数通过 ScanConfig 配置类传入，支持独立设置上下颌参数。
maxillaTypes，int 类型（位或组合），上颌修复类型组合，可多选组合，异性杆优先级最高
mandibleTypes，int 类型（位或组合），下颌修复类型组合，与上颌规则一致
occlusionType，EOcclusionType类型，咬合类型，仅自然咬合 / 咬合记录两种
isMaxillaTemporary，bool 类型，上颌临时牙开关，控制上颌是否按临时牙逻辑生成流程
isMandibleTemporary，bool 类型，下颌临时牙开关，控制下颌是否按临时牙逻辑生成流程
isFullMouthTemporary，bool 类型，全口临时牙开关，优先级最高，开启后上下颌均强制按临时牙处理
isMaxillaRod2Enable，bool 类型，上颌第二扫描杆开关，同时控制普通扫描杆 2、异性扫描杆 2 的显示
isMandibleRod2Enable，bool 类型，下颌第二扫描杆开关，与上颌规则一致
beforePrep，EBeforePrep类型，备牙扫描开关，仅「仅修复」场景生效
四、详细业务规则
4.1 通用规则
整体流程顺序：上颌扫描位点 → 【可选：数据交换位点】 → 下颌扫描位点 → 咬合扫描位点
临时牙优先级：全口临时牙 > 单颌临时牙；全口临时牙开启时，单颌设置失效
备牙生效条件：仅在「仅修复（无种植、无异性杆）」场景下生效；存在种植或异性杆时不生成备牙位点
位点展示规则：临时牙场景复用自然牙按钮展示，界面不单独显示「临时牙」按钮
分支优先级：异性扫描杆 > 种植 > 仅修复 / 无修复
4.2 单颌扫描流程规则（上下颌完全对称）
以下以上颌为例，下颌逻辑完全一致，仅枚举名称替换为对应下颌枚举。
场景 1：存在异性扫描杆
核心规则：无论是否开启临时牙，异性杆流程必须包含袖口位点。
开启临时牙时流程：自然牙上颌 → 异性扫描杆 1 上颌 → 【异性扫描杆 2 上颌】 → 袖口上颌
未开启临时牙时流程：异性扫描杆 1 上颌 → 【异性扫描杆 2 上颌】 → 袖口上颌
注：扫描杆 2 开启时才生成对应位点。
场景 2：无异性杆、存在种植
核心规则：仅临时牙种植场景生成袖口位点，普通种植场景无袖口。
开启临时牙时流程：自然牙上颌 → 袖口上颌 → 扫描杆 1 上颌 → 【扫描杆 2 上颌】
未开启临时牙时流程：自然牙上颌 → 扫描杆 1 上颌 → 【扫描杆 2 上颌】
场景 3：仅修复 / 无修复（无种植、无异性杆）
核心规则：临时牙与非临时牙均复用自然牙按钮，备牙按需追加。
开启临时牙时流程：自然牙上颌 → 【备牙上颌】
未开启临时牙时流程：自然牙上颌 → 【备牙上颌】
注：备牙开启时才生成备牙位点。
4.3 咬合扫描规则
自然咬合 → 生成「自然牙咬合」位点
咬合记录 → 生成「咬合记录」位点
注：临时牙开关不影响咬合位点生成。
4.4 数据交换（交换按钮）生成规则
满足以下任意一条时，不生成数据交换位点；所有规则都不满足时才生成：
上颌或下颌存在异性扫描杆
备牙功能开启
上颌扫描杆 2 开启
下颌扫描杆 2 开启
任意临时牙开关开启（上颌 / 下颌 / 全口临时牙）
4.5 下一个扫描位点查询规则
按「上颌→下颌→咬合」顺序遍历完整流程
自动跳过数据交换部分所有位点，不纳入下一位点计算
当前位点不存在或已是最后一个时，返回无效位点 ScanSite_None
位点匹配严格按枚举值相等判断

六、典型场景验证用例
用例 1：纯自然普通修复（生成交换按钮）
配置：
上颌：仅修复，无临时牙，无杆 2，无备牙
下颌：仅修复，无临时牙，无杆 2，无备牙
咬合：自然咬合
预期流程：
上颌：自然牙上颌 → 交换按钮 → 下颌：自然牙下颌 → 咬合：自然牙咬合
下一位点查询结果：自然牙上颌 → 自然牙下颌（自动跳过交换）
用例 2：上颌异性杆 + 下颌种植临时牙（不生成交换按钮）
配置：
上颌：修复 + 异性杆，无临时牙，无杆 2
下颌：仅种植，开启下颌临时牙，无杆 2
备牙：关闭
预期流程：
上颌：异性扫描杆 1 上颌 → 袖口上颌
下颌：自然牙下颌 → 袖口下颌 → 扫描杆 1 下颌
咬合：自然牙咬合
无数据交换按钮
用例 3：仅修复 + 备牙（不生成交换按钮）
配置：
上下颌均为仅修复，开启备牙，无临时牙
咬合：自然咬合
预期流程：
上颌：自然牙上颌 → 备牙上颌
下颌：自然牙下颌 → 备牙下颌
咬合：自然牙咬合
无数据交换按钮
七、代码优化方向建议
规则配置化：将业务规则抽象为 JSON/XML 配置表，无需修改代码即可调整流程顺序与分支
状态机改造：封装为有限状态机，支持更复杂的跳转、回退、跳过与异常处理逻辑
强类型枚举：使用 enum class 替代普通枚举，增强类型安全性与编译期检查
缓存优化：配置不变时缓存流程结果，避免重复计算，提升界面渲染效率
单元测试：补充全场景单元测试，覆盖所有边界条件与规则组合
错误处理：增加非法参数校验、日志输出与降级策略，提升模块健壮性
