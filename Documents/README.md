## 维护顺序

1. 后续任务开始前，先读取本目录四份核心文档和目标模块自己的 `README.md` / `CHANGELOG.md`。
2. 方案变化先修改本目录权威文档，再同步到 `D:\wj\重构文档` 的同名镜像；禁止分别编辑后再人工猜测冲突。
3. 模块行为变化同时更新模块内 `README.md` / `CHANGELOG.md`，并以中文详细提交信息提交。
4. 提交前运行注释安全检查、受影响模块测试、MainExe 主链路 smoke 和 `git diff --check`。
5. GitHub 提交完成后，使用 `tools\BackupToLocalRepository.ps1` 整体同步到本地仓库；不得只备份单个模块。

## 现行内容阅读顺序

1. 先读《重构任务总览》末尾最新日期章节，确认产品范围、全局约束和当前成熟度，不能只看早期模块草案。
2. 再读《架构设计与接口规范》第 14 章；若第 5-6 章历史接口草案与第 14 章冲突，以第 14 章和仓库公共头文件为准。
3. 再读《开发进度跟踪》顶部“最新验证基线”和“当前未完成闭环”，区分“框架已接通”“smoke 已通过”“真实业务已完成”。
4. 最后按日期查《AI 协作记录》，了解某条规则形成原因；历史记录只能用于追溯，不能覆盖更新日期更晚的现行规则。

## 统一验收入口

- 根 CMake 构建后执行 `ctest --test-dir F:\MeyerScan\build -C Release -N`，当前应列出 24 项测试。
- 执行 `ctest --test-dir F:\MeyerScan\build -C Release --output-on-failure`，统一运行模块回归和 MainExe 集成 smoke。
- CTest 通过不替代三档 UI 截图、真实数据库迁移、设备/算法、长时间稳定性和安装包验收；这些仍按进度文档中的未完成项单独验证。
