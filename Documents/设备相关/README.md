# 设备相关文档

本目录是 MeyerScan 设备通信、机型识别、设备信息保存和通信协议的知识库。它服务于开发人员、测试人员、现场支持人员和后续参与项目的 AI，重点记录“为什么这样设计、代码如何调用、哪些结论已经验证、哪些结论不能假设”。

## 文档入口

| 文档 | 内容范围 |
|---|---|
| `口扫各系列机型使用指南.md` | 产品系列和具体型号、设备识别流程、生产模式、颜色校准准入、代码调用注意事项、调试和排错方法 |
| `口扫设备协议文档.md` | A/B 类帧、公共命令、机型专用命令、完整帧示例、字段定义、错误分类和验证状态 |

## 与代码的对应关系

- `MyDeviceTransport`（输出 `MeyerScan_DeviceTransport.dll`）：负责 USB 连接、USB 速率、原始命令字节收发、异步传输和数据帧交付，不解释业务命令含义。
- `MyDeviceCmd`（输出 `MeyerScan_DeviceCmd.dll`）：负责命令编码、响应查找、长度/命令码/求和校验、回包解析、型号目录和设备检测快照。
- `MyMainExe`：持有进程内唯一设备会话，将设备预检结果注入工作台、设置和校准流程，负责创建/练习模式的准入策略。
- `MySettingsUI`、`MyCalibrationColorUI`、扫描界面：只接收固定 POD 快照或宿主动作结果，不直接打开 USB、不加载 DeviceTransport、不解析原始回包。

修改设备协议代码前，先阅读本目录指南和协议文档，再阅读 `MyDeviceCmd/docs/ProtocolCommandCoverage.md`、`MyDeviceCmd/include/DeviceCmd.h` 以及目标模块 README/CHANGELOG。代码实现、协议文档和测试结果必须保持同一结论。

## 维护规则

1. 文档使用中文维护；代码注释和模块 CHANGELOG 使用中文；Git 提交信息使用中文并详细说明变更。
2. 新增机型先增加产品目录、协议 Profile 和测试数据，再接入 UI 或工作流。不能在 UI 中复制一份设备型号判断表。
3. 每条结论都区分“协议资料确认”“模拟测试通过”“实机测试通过”和“待验证”。模拟后端通过不等于真实设备通过。
4. 设备真实上报值和流程兼容使用的有效值必须分别记录。生产设备的默认编号/型号只能在文档规定的练习、颜色校准或三维校准场景使用，不能覆盖真实字段。
5. 每个协议版本必须写明适用机型、命令范围和验证状态；只覆盖一个机型的协议不能直接套用到全系列。
6. 每次命令、型号映射、时序或错误分类变化，同时更新本目录文档、`MyDeviceCmd` 的覆盖表、测试项目和 CHANGELOG。
7. 本目录不保存客户隐私、真实设备授权密钥或未经确认的生产数据。示例编号只能使用脱敏数据或代码中已有的模拟值。
8. 文档随全部模块一起提交 GitHub，并进入 `F:\MeyerScan-Reposit` 整体备份；不能只提交设备文档而漏掉对应代码和测试。

## 协议维护约定

后续补充协议时，在 `口扫设备协议文档.md` 中按“公共命令、机型专用命令、帧字段、完整示例、错误分类、模拟验证、实机验证”的固定结构维护。新增协议不得覆盖既有机型结论；涉及客户信息或生产数据时必须先脱敏。
