# 口扫各系列机型使用指南

本文是开发和调试指南，不是面向最终客户的操作手册。它把产品型号、协议 Profile、代码边界和常见实现技巧放在一起，帮助后续开发人员或 AI 快速理解设备链路。未知内容明确标记为“待定”或“待验证”，不得根据相似名称自行补全。

> 重构软件正式适配范围只有 `mOS MyScan 5` 和 `mOS MyScan 6`。旧 `mOS MyScan` 已停产且用户少，保留协议、产品目录和诊断代码用于识别及维护，但检测到该系列后必须提示“当前软件不支持该系列设备”，不能进入校准、创建或练习扫描。

## 1. 名称层级

设备识别必须区分三层：

1. **协议/硬件能力 Profile**：例如 `MyScan3`、`MyScan5`、`MyScan5H`、`MyScan6`、`MyScan6Wireless`。它决定 USB、命令、采集和版本读取差异，不直接作为产品展示名称。
2. **产品系列**：`mOS MyScan`、`mOS MyScan 5`、`mOS MyScan 6`。它决定颜色校准扫描头策略等业务规则。
3. **具体产品型号**：例如国内标准版、P1/P2/P3、国外版、医院版。它由完整型号代码和设备编号前缀等证据综合确定。

UI、MainExe 和业务模块不得重新编写这三层之间的映射；统一使用 `MyDeviceCmd` 返回的 `MeyerDeviceProductIdentity` 和 `MeyerDeviceDetectionRecord`。

## 3. 设备信息保存方式

设备信息由 DeviceCmd 生成固定宽度 POD 快照，再由宿主缓存和转发：

- `reported*`：设备真实回包解析出的编号、型号代码、系列、版本和状态。没有回包时保持“未知”，不能用空字符串或数值 0 假装读取成功。
- `effective*`：当前流程允许使用的有效身份。练习、颜色校准和三维校准可以在文档规定的生产模式下使用带来源的兼容默认值；创建模块进入扫描前必须有真实设备编号。
- `source`、`productionMode`、`compatibilityUsed`、各步骤状态和原始诊断：用于说明有效值从哪里来，防止旧固件兼容逻辑被误认为设备真实上报。
- MainExe 的 `DeviceSessionHost` 是进程内唯一设备会话所有者。SettingsUI、CalibrationColorUI、ScanWorkflowUI 只读取不可变状态快照，不复制 USB 句柄。

典型读取关系如下：

```text
MyDeviceTransport -> MyDeviceCmd -> DeviceSessionHost -> MainExe
                                           |             |
                                           +-> POD 快照 -> SettingsUI / CalibrationColorUI / ScanUI
```

如果 UI 需要设备编号，应读取宿主注入的快照字段，而不是再次执行 `0xD4`，也不能从日志文本反向解析。

## 4. 设备型号识别流程

### 4.1 常规链路

颜色校准、设置页设备信息和进入扫描前检查共用只读预检原则，实际编排由宿主完成：

```text
打开设备
  -> 检查 USB 连接和 USB2/USB3
  -> 0xD4 请求 / 0xD9 回包：读取 13 位设备编号
  -> 若编号未写入：0xC2 / 0xC7 探测生产设备的系列候选
  -> 0xCD 请求 / 0xCE 回包：读取设备信息和型号代码
  -> ProductCatalog 综合编号前缀、完整型号代码和 Profile
  -> 产品系列支持门禁：旧 mOS MyScan 返回不支持并关闭会话
  -> 0x14 / 0x15：读取主控板版本
```

`0xD4/0xD9` 在公共 API 中仍有历史名 `MachineCode`，产品含义统一写作“设备编号”。`0xCD/0xCE` 的回包布局不能跨机型按同一偏移解释：旧有线实例与 MyScan 6 Wireless 的字段布局不同，必须由 Profile 选择解析器。

### 4.2 生产模式和未写编号

新设备可能在生产调试期间没有设备编号，直到最后打包环节才写入。D9 命令出现以下两种情况时，都表示“设备编号未写入”，但必须分别保存原因：

- 命令确认是 `0xD9`，长度字段为 `0xFFFF`：记录 `UninitializedLength`。
- 命令确认是 `0xD9`，求和校验失败：记录 `ChecksumIndicatesUnprogrammed`。

未收到 D9、普通坏帧、错误响应码、长度截断或回包无法确认是 D9 时，不能直接判定为生产模式，只能记录对应回包异常。错误帧和生产未写号必须在字段上区分。

受支持系列的生产模式默认身份只用于：

- 练习模块：默认允许，受 ConfigCenter 的生产模式开关控制。
- 颜色校准和后续三维校准：不要求正式编号，但仍需完成型号/能力/版本等准入。

创建模块：默认不允许。没有真实设备编号时，必须停留在建单或前置检查阶段，不能创建 Scan、Process、Send 页面，也不能把兼容编号写进正式订单。

## 5. 颜色校准设备规则

- `mOS MyScan`：保留“大扫描头参数供大小扫描头共用”的旧协议代码和测试，但重构软件在产品系列门禁处已经拦截，不再发送 `0x14/0x15`、`0x12/0x13`、`0xA3/0xA4` 或进入校准 UI。
- `mOS MyScan 5/5H/6`：主控板版本必须满足 `1.3.x` 或更高；`1.1.x`、`1.2.x` 和无法解析的版本保守拦截。通过版本门禁后分别读取大扫描头 `0xA3/0xA4` 和小扫描头 `0xB9/0xBA`。
- 期望响应码正确但求和失败：对应扫描头记录为“未校准”，可以按流程提示用户。
- 无回包、错误帧头、错误响应码、截断或 payload 长度错误：记录为“读取失败”，禁止把它显示成“未校准”并继续进入。
- 大小扫描头状态应保存在 `MeyerDeviceScanHeadColorCalibrationSnapshot`，由 MainExe/SettingsUI/CalibrationColorUI 逐字段复制，UI 不解析原始矩阵。

提示规则：大小都已校准不提示；仅大头未校准提示“大扫描头未校准”；仅小头未校准提示“小扫描头未校准”；两者都未校准提示“大、小扫描头都未校准”。界面可见文本仍遵守 `tr("English source text")` 规则，由翻译资源提供中文显示。

## 6. 代码调用与实现技巧

### 6.1 DLL 边界

DeviceTransport 和 DeviceCmd 是非 Qt 设备基础模块，公共接口使用 C ABI 和固定布局结构：

- 结构体第一字段包含 `structSize`、`schemaVersion` 或等价 ABI 信息，并保留 `reserved` 字段。
- 跨 DLL 只传固定宽度整数、固定长度 UTF-8 数组、调用方提供的缓冲区和原始字节。
- 禁止跨边界传 `std::string`、`std::vector`、Qt 容器、`QObject`、USB 句柄或异常对象。
- DLL 内部可以使用 C++ 字符串和容器，因为内存所有权不会越过边界；导出函数必须把结果复制到调用方结构体。

这种设计的关键技巧是“调用方分配、被调用方填充”：调用方先调用初始化函数获得正确的 schema 和默认值，再传入结构体；被调用方只写入容量范围内的字段。扩展字段时优先在结构体末尾追加，并提升 schema；不能在中间插入字段改变旧布局。

### 6.2 绝对路径和动态加载

DeviceCmd 运行时动态加载 `MeyerScan_DeviceTransport.dll`，路径必须由宿主根据 EXE/DLL 实际目录生成。Qt 宿主使用 `QCoreApplication::applicationDirPath()`，纯 C++ 模块使用 Windows 模块路径接口；禁止使用 `QDir::currentPath()`、进程当前工作目录或开发机绝对路径。

加载过程要逐步记录：绝对路径、LoadLibrary/QLibrary 结果、导出函数地址、API 版本、初始化结果和关闭结果。DLL 能加载不代表 API 版本匹配或模块可用，缺少导出函数、版本不匹配和 Init 失败都必须中止该链路。

### 6.3 命令帧和回包解析

一个命令通常由“请求码 + 固定协议头 + payload 长度 + payload + 求和/尾部校验”组合而成；具体字段以协议登记和代码帧解析器为准。读取设备编号时，DeviceCmd 发送 `0xD4`，从传输层取得原始回包，先确认帧头、响应码 `0xD9`、长度和求和，再把 13 个数字复制到 `MeyerDeviceMachineCode`。任何一步失败都应返回细分状态，不能只返回一个“读取失败”。

回包解析应返回固定结果结构，而不是 `string` 或 `string[]`：结构体可同时保存状态码、期望/实际命令码、长度、校验结果、原始 payload 长度和已解析字段。调试工具可以把结构体格式化成文本；业务模块读取结构化字段，不从文本猜测状态。

### 6.4 命令时序

当前实现约束如下：

| 场景 | 当前规则 |
|---|---:|
| 普通低频命令回包超时 | 200 ms |
| 上一条命令没有可识别回包时，下一条命令前兜底间隔 | 20 ms |
| D4/D9 发送后到接收 | 50 ms |
| CD/CE 发送后到接收 | 50 ms |
| MyScan 3 主控板到投图板切换 | 20 ms，仅 Profile 特例 |
| 图像流超时 | 1500 ms，独立于低频命令 |

命令 A 收到并解析出期望回包后，不再额外等待 20 ms；20 ms 只是上一条没有完成响应时的兜底。不要在 UI 线程中 `Sleep`，命令队列和设备会话应由专用工作线程串行执行。

## 7. 调试、日志和测试

建议按以下顺序排查：

1. 确认运行目录中的 DeviceTransport/DeviceCmd DLL、依赖库和版本清单齐全。
2. 运行 DeviceTransport smoke，确认枚举、USB 速率、打开和关闭。
3. 运行 DeviceCmd smoke，确认命令编码、模拟回包和错误分类。
4. 使用真实设备预检，逐项查看 Open、USB、D4-D9、C2-C7、CD-CE、ProductCatalog、Firmware 和 Total 耗时。
5. 检查日志中的请求码、响应码、长度、求和、解析状态、reported/effective 值和来源；不要只看最终“Ready”。
6. 分别验证“无设备、USB2、D9 无回包、D9 0xFFFF、D9 求和失败、CE 无回包、错误版本、扫描头未校准、创建模式生产设备”等门禁。

设备层日志应记录关键命令和错误，但不能逐帧或逐次 `NotReady` 形成日志风暴。真实设备结果必须注明机型、连接方式、固件版本、测试时间和测试结论；模拟测试不得写成实机成功。

VS2015 常见问题是 x86/x64、Qt 版本和运行时 DLL 不匹配。CMake、VS2015 和 VSCode 必须使用同一架构和同一套 Qt/第三方库；不要把开发机绝对路径写入项目文件。构建输出目录不要被 VS2015 与 CMake 并行占用。
